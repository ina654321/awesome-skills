## 5. Implementation Guidelines / 实施指南

### 5.1 Section Quality Standards / 章节质量标准

#### §3 Risk Disclaimer — How to Write 9.5/10
<!-- §3 风险提示 — 如何写到 9.5/10 -->

```markdown
❌ BAD (2/10 — Generic, no domain specificity):
| Risk | Severity | Description | Mitigation |
|------|----------|-------------|------------|
| AI may be wrong | 🔴 High | AI errors | Verify output |

✅ GOOD (9.5/10 — Domain-specific, concrete consequences):
| Risk / 风险 | Severity / 严重度 | Description / 描述 | Mitigation / 缓解措施 |
|------------|-----------------|-------------------|---------------------|
| **Wrong DB for workload** | 🔴 High | Choosing MongoDB for financial transactions breaks ACID guarantees, causing data inconsistency under concurrent writes | Audit read/write ratio first; use PostgreSQL for any multi-entity transactions; load test before committing to schema |
| **N+1 query in production** | 🔴 High | ORM lazy loading fetches 1 query per row → 1000 users = 1001 DB queries → 10s page load | Use EXPLAIN ANALYZE in staging; enforce eager loading with `.include()` in code review |
```

#### §8 Standard Workflow — How to Write 9.5/10
<!-- §8 标准工作流程 — 如何写到 9.5/10 -->

```markdown
❌ BAD (3/10 — Abstract, no checkpoints):
1. Gather requirements
2. Design solution
3. Implement
4. Test

✅ GOOD (9.5/10 — Phased with templates and Done criteria):
Phase 1: Discovery & Assessment (Day 1-2)
├── Define constraints: SLA, budget, team size, tech debt
├── Document current architecture (draw.io or C4 diagram)
├── Identify top-3 bottlenecks via profiling or stakeholder interviews
└── [✓ Done]: 1-pager with problem statement + success metrics signed off

Phase 2: Design & Planning (Day 3-5)
├── Produce 3 solution options with trade-off matrix
├── Run Architecture Decision Record (ADR) for each major choice
├── Create implementation backlog with T-shirt sizing
└── [✓ Done]: ADR approved by tech lead; story points estimated

Phase 3: Execution & Delivery (Week 2-4)
├── Build → test → code review → staging deploy (daily cadence)
├── Gate: automated test coverage ≥80% before each merge
└── [✓ Done]: prod deploy passes all SLOs for 48h; rollback tested
[✗ FAIL Action]: Escalate to architect if p99 >2× baseline after 24h
```

#### §10 Common Pitfalls — How to Write 9.5/10
<!-- §10 常见陷阱 — 如何写到 9.5/10 -->

```markdown
❌ BAD (2/10 — Generic, not domain-specific):
- Don't use the wrong tool
- Always test your code

✅ GOOD (9.5/10 — Named anti-patterns with concrete consequences):
**Anti-Pattern: The God Service / 上帝服务**

❌ BAD: Building one "OrderService" that handles orders, inventory, payments,
notifications, reporting, and analytics (15,000 LOC, 200ms p99, 4 deploys/day
because everything coupled).

✅ GOOD: Split by business capability: OrderService (create/cancel),
InventoryService (reserve/release), PaymentService (charge/refund).
Each deploys independently → team A can ship payment fixes without
blocking team B's order features.

**Why it matters:** God services become organizational bottlenecks. When
6 teams all modify one service, merge conflicts alone cost 20% of sprint
velocity. Conway's Law: architecture mirrors org structure.
```
