# Glossary

## dbt Core Concepts

### Model
A `.sql` file in the `models/` directory that defines a SELECT statement. Models are compiled into SQL and run against the data warehouse. Each model represents a table, view, or incremental model.

### Materialization
How dbt stores model results in the database. Options: `view`, `table`, `incremental`, `ephemeral`, `external`. Chosen via the `{{ config() }}` block.

### Ref
The `ref()` function references other models within the dbt project. It automatically handles dependencies and computes the correct schema/table name. Usage: `{{ ref('model_name') }}`.

### Source
Raw data from external systems, defined in `sources.yml`. Accessed via `source('source_name', 'table_name')`. Enables documentation, testing, and freshness checks on upstream data.

### Seed
A CSV file in the `seeds/` directory loaded as a static table via `dbt seed`. Used for small, infrequently changing reference data (e.g., country codes, product taxonomy).

### Macro
Reusable Jinja code stored in `macros/`. Can be called as functions with `{{ my_macro(arg1, arg2) }}`. Used for repetitive SQL patterns, cross-database compatibility, and custom logic.

### Package
An external dbt project installed via `packages.yml`. Contains models, macros, and tests. Popular packages: `dbt-utils`, `codegen`, `audit_helper`.

### Test
A validation check on a model or column. Two types:
- **Schema tests** (generic): `unique`, `not_null`, `relationships`, defined in `schema.yml`
- **Custom tests** (singular): custom SQL queries returning zero rows for pass

### Generic Test
A reusable test defined in `tests/generic_tests/` or a package. Defined once, applied to many columns via `tests:` key in `schema.yml`.

### Custom Test
A one-off SQL query in `tests/` directory or inline in `schema.yml` that returns failing rows.

### Hook
SQL executed before or after a model runs. `pre-hook` runs before, `post-hook` runs after. Used for grants, table swaps, or DDL operations.

### Variable (var)
A project-level or CLI-overridable value set in `dbt_project.yml` or via `--vars`. Used for environment-specific configuration. Access via `{{ var('var_name') }}`.

### Target
The active database connection profile (dev, prod, etc.). Set via `DBT_TARGET` env var or `--target` CLI flag.

### Manifest
A JSON file (`target/manifest.json`) generated by `dbt compile`. Contains metadata about all models, sources, tests, and their relationships. Used by docs, lineage graphs, and the dbt Cloud DAG.

### Contract
A schema enforcement feature (dbt 1.8+) that defines expected column names, types, and constraints. Breaks the build if the actual schema doesn't match the contract.

### Semantic Layer
dbt's framework for defining metrics as code. Metrics are defined in `metrics:` blocks and exposed via the dbt Semantic Layer API for consumption by BI tools.

### Metric
A named business measure (e.g., `revenue`, `order_count`) defined in `schema.yml` or `metrics:` blocks. Metrics reference dimensions and can be combined.

### Exposure
A definition of a downstream consumer of dbt models (dashboard, report, application). Exposed in the dbt DAG for lineage tracking and impact analysis.

### Slim CI
A dbt Cloud feature (and CLI pattern) that only runs models/tests affected by code changes, using the `state:modified` selector to compare against a previous manifest.

### Adapter
A plugin that connects dbt to a specific data platform. Examples: `dbt-snowflake`, `dbt-bigquery`, `dbt-databricks`, `dbt-postgres`, `dbt-spark`.

### Cross-database Macro
Macros (e.g., `date_diff`, `date_trunc`) that work across multiple adapters by dispatching to adapter-specific implementations. Found in `dbt-core` and `dbt-utils`.

### Incremental Strategy
Defines how dbt handles new rows in incremental models:
- `append` (default): insert all rows
- `merge`: upsert using `unique_key`
- `delete+insert`: delete matched rows, insert new

### Dispatch
The mechanism where generic macros in `dbt.internal` are overridden by adapter-specific implementations. Set via `macro__dispatch` pattern in custom macros.

### Node Selection Syntax
dbt's selector language for targeting specific models:
- `model_name` ŌĆö exact match
- `+model` ŌĆö all parents (upstream)
- `model+` ŌĆö all children (downstream)
- `tag:staging` ŌĆö by tag
- `path:models/marts/` ŌĆö by path
- `@source:stripe` ŌĆö lineage from source

## CLI Reference

| Command | Purpose |
|---------|---------|
| `dbt init` | Initialize a new project |
| `dbt debug` | Validate project configuration |
| `dbt deps` | Install packages |
| `dbt run` | Execute models |
| `dbt test` | Run schema and data tests |
| `dbt build` | Run + test + seed combined |
| `dbt compile` | Generate SQL without running |
| `dbt docs generate` | Build documentation |
| `dbt docs serve` | Local documentation server |
| `dbt ls` | List selected nodes |
| `dbt seed` | Load CSV seeds |
| `dbt source freshness` | Check upstream freshness |
| `dbt clean` | Remove build artifacts |
| `dbt retry` | Retry last failed run |
| `dbt run --full-refresh` | Force full materialization |
| `dbt run --select +model` | Run upstream dependencies |
| `dbt run --vars '{"k":"v"}'` | Pass variables |
