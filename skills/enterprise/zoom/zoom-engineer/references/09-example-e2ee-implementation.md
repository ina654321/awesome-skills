# Example: End-to-End Encryption Implementation

## Scenario

**Context:** Security researchers and enterprise customers raised concerns about Zoom's encryption model. Standard transport encryption (TLS/DTLS) means Zoom servers can technically access meeting content. Enterprise customers demand true end-to-end encryption (E2EE) where only participants can decrypt content.

**Business Impact:**
- Security researchers published encryption gap analysis
- Enterprise customers delaying renewals
- Regulatory scrutiny increasing
- Competitive pressure from E2EE-native products

## Requirements

### Security Goals

| Requirement | Description |
|-------------|-------------|
| **True E2EE** | Zoom servers cannot decrypt meeting content |
| **Key Verification** | Participants can verify each other's keys |
| **Backward Compatibility** | Optional feature, not breaking existing workflows |
| **Performance** | Latency impact <20ms |
| **Usability** | No security expertise required from users |

### Threat Model

```
Threats E2EE Protects Against:
├── Zoom employees accessing meeting content
├── Government subpoenas for meeting data
├── Data center compromises
├── Insider attacks
└── Cloud provider access

Threats E2EE Does NOT Protect Against:
├── Participant devices compromised
├── Screen capture by participants
├── Meeting links shared inappropriately
└── Post-meeting data leakage
```

## Cryptographic Design

### Key Exchange Protocol

```
E2EE Key Hierarchy:

┌─────────────────────────────────────────────────────────┐
│                   MEETING LEVEL                          │
│  Meeting Key (MK) - Per-meeting symmetric key           │
│  Generated by host device                                │
│  Distributed via public key encryption                   │
├─────────────────────────────────────────────────────────┤
│                 PARTICIPANT LEVEL                        │
│  Identity Keys: Curve25519 key pairs                    │
│  • Long-term identity key (persistent)                  │
│  • Ephemeral session key (per-meeting)                  │
├─────────────────────────────────────────────────────────┤
│                  STREAM LEVEL                            │
│  Stream Keys: Derived from MK via HKDF                  │
│  • Audio stream key                                     │
│  • Video stream key                                     │
│  • Screen share key                                     │
└─────────────────────────────────────────────────────────┘
```

### Cryptographic Primitives

| Component | Algorithm | Purpose |
|-----------|-----------|---------|
| **Key Exchange** | X25519 (Curve25519) | Ephemeral key agreement |
| **Digital Signatures** | Ed25519 | Identity verification |
| **Symmetric Encryption** | AES-256-GCM | Media stream encryption |
| **Key Derivation** | HKDF-SHA256 | Deriving stream keys |
| **Hash** | SHA-256 | Integrity verification |

```
    │                       │                       │
    │  2. Get Bob's public  │                       │
    │     key from server   │                       │
    │──────────────────────→│                       │
    │                       │──────────────────────→│
    │                       │←──────────────────────│
    │◄──────────────────────│  (Bob's identity key) │
    │                       │                       │
    │  3. Encrypt MK with   │                       │
    │     Bob's public key  │                       │
    │◄──────────────────────┤                       │
    │                       │                       │
    │  4. Send encrypted MK │                       │
    │     to Bob via server │                       │
    │──────────────────────→│──────────────────────→│
```

## Technical Implementation Details

### Media Encryption

```c
// Simplified E2EE encryption flow

// 1. Derive stream key from meeting key
uint8_t stream_key[32];
hkdf_sha256(
    meeting_key, 32,           // input key material
    (uint8_t*)"audio-stream",  // context info
    12,                        // context length
    stream_key, 32             // output
);

// 2. Encrypt RTP packet
void encrypt_rtp_packet(rtp_packet_t* packet, uint8_t* stream_key) {
    // Generate nonce (IV)
    uint8_t nonce[12];
    generate_nonce(nonce, packet->sequence_number);
    
    // Encrypt payload with AES-256-GCM
    uint8_t ciphertext[1500];
    uint8_t tag[16];
    aes_256_gcm_encrypt(
        stream_key,              // key
        nonce,                   // IV
        packet->payload,         // plaintext
        packet->payload_len,     // plaintext len
        packet->header,          // additional data
        packet->header_len,      // additional data len
        ciphertext,              // output
        tag                      // auth tag
    );
    
    // Replace payload with ciphertext + tag
    memcpy(packet->payload, ciphertext, packet->payload_len);
    memcpy(packet->payload + packet->payload_len, tag, 16);
    packet->payload_len += 16;
}
```

```
┌─────────────────────────────────────────────────────┐

│  │    12345  67890      │   │    12345  67890      ││

│  └──────────────────────┘   └──────────────────────┘│

│                                                     │
└─────────────────────────────────────────────────────┘

Safety Number = SHA-256(
    host_identity_key || participant_identity_key || meeting_id
)
```

### Server Role (Relay Only)

```
Server Cannot:
├── Decrypt meeting keys (encrypted to participants)
├── Access media content (E2EE encrypted)
├── Decrypt chat messages
└── Access screen share content

Server Can:
├── Relay encrypted packets (opaque forwarding)
├── Manage meeting state (participants, permissions)
├── Route key distribution messages
└── Track metadata (duration, participants count)
```

## Key Lessons

1. **Transparency Builds Trust**
   - Publishing design for peer review was crucial
   - Acknowledging limitations honestly

2. **Usability vs Security Balance**
   - Phone verification for anti-abuse
   - Safety numbers for verification without complexity
   - Clear UI showing what's protected

3. **Gradual Rollout**
   - Beta allowed iteration
   - Phone verification prevented spam
   - Feature flags enabled quick rollback

4. **Acquire Expertise**
   - Keybase acquisition brought cryptographers
   - Existing code accelerated development
   - Credibility with security community
