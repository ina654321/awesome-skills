## § 3 · Risk Framework

### Critical Risks (Nintendo-Specific)

| Risk | Severity | Mitigation | Escalation |
|------|----------|------------|------------|
| **Tech-First Design** | 🔴 Critical | Paper prototype 2 weeks before code | Miyamoto design review |
| **Complexity Creep** | 🔴 Critical | Freeze mechanics at vertical slice; new ideas → sequel | Producer scope review |
| **Accessibility Failure** | 🔴 Critical | Test with non-gamers; add Super Guide if 90% fail | Kid-test failure protocol |
| **Hardware Underutilization** | 🟡 High | Dev kit exploration phase mandatory | Hardware co-design session |
| **Feature Creep** | 🟡 High | "No new mechanics in production" rule | EPD Director approval |
| **Patch Culture Temptation** | 🟠 Medium | Ship-it-ready standard; no day-one patches | Quality assurance gate |
| **Trend Chasing** | 🟠 Medium | "Would this exist without [trend]?" test | Strategic planning review |

### Risk Response Playbooks

**Scenario: Kid-Test Failure**
```
Trigger: <50% of 5-year-olds complete tutorial without help
Response:
1. Halt production on affected section
2. Observational study: Where exactly do they stop?
3. Simplify inputs or visual language
4. Retest with new cohort
5. If still failing: Full redesign from fundamentals
```

**Scenario: Deadline vs. Quality Conflict**
```
Trigger: Game not meeting "delicious" standard at milestone
Response:
1. Document specific quality gaps
2. Present to leadership with delay request
3. Reference: "A delayed game is eventually good..."
4. Historical precedent: Zelda: Breath of the Wild (delayed 2 years)
5. Secure additional development time
```
